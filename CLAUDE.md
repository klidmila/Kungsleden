# Kungsleden Companion — architecture & build spec

This file is project context for an AI agent (or a human) picking up work on
this repo. `readme.md` covers what the app is and how to run it day to day;
this file explains *why it's built the way it is* and the rules that keep it
that way. It replaces the old `pilot_brief.md`, which specified a small
two-stretch demo pilot -- that scope is gone, the app now covers the full
route, and this file documents the result rather than the original demo.

## Goal

An offline-first travel companion for hiking the Kungsleden trail (Kvikkjokk
to Abisko, south to north). The single hard requirement everything else
follows from: **it has to keep working with no signal in the mountains, from
a cold start, with nothing pre-cached.**

## Non-negotiable constraint: zero network dependency, always

That requirement rules out anything that needs a live request to resolve,
even a same-origin one:

- No runtime `fetch()` of local JSON (blocked by CORS under `file://`; even
  served over HTTP it still needs a live round-trip on every load, which
  fails offline without an explicit cache)
- No `<script type="module">` (module scripts are also blocked by CORS under
  `file://` in Chromium browsers -- same failure mode as `fetch()`, easy to
  miss because it only shows up when someone opens the file directly instead
  of through a dev server)
- No CDN scripts, external fonts, or analytics

Consequence: `index.html` is one self-contained file -- HTML shell + CSS +
JS + all trip data inlined in a single `<script>` block. Opening it via
`file://` (e.g. a saved file on a phone with no signal) must work with zero
network activity, full stop. An earlier version of this app fetched JSON at
runtime (a cleaner-looking SPA split across `js/data.js`/`render.js`/
`main.js`); it was reverted specifically because it broke this requirement.
Don't reintroduce runtime fetch or ES modules without re-solving offline
loading first (e.g. a service worker that's actually wired up and tested,
not just planned).

## Architecture

- `index.html` is a **build output**. The data block between the
  `// ===== GENERATED DATA: START =====` / `END` markers (`waypoints`,
  `alertsData`, `crossings`, `legs`) is generated by `build.js`. Everything
  else in the file -- shell markup, CSS, render/nav JS -- is hand-authored
  and untouched by the build.
- Never hand-edit content between those markers; `node build.js` overwrites
  it. Edit the source JSON files instead, then rebuild.
- `build.js` (Node, no dependencies) is the only build step, and the only
  thing that reads the JSON files at all -- once `index.html` is generated,
  it has no connection back to them. It replaced an earlier PowerShell
  script (`generate.ps1`, removed) that did the same job for a hand-picked
  2-segment subset of the route; the Node version covers the full route and
  is driven entirely by `crossings.json` (see below) instead of a hardcoded
  list.
- `build.js` is idempotent: unchanged JSON input regenerates a
  byte-identical `index.html`.

## Data files & id scheme

All source data lives under `data/`. `route.json` holds every waypoint and
segment for the full route. `alerts.json` holds trip-critical warnings.
`crossings.json` is a manifest mapping each `boat_*.json`/`bus_*.json`
crossing file to the segment(s) it belongs to -- the one piece `build.js`
can't derive on its own, since nothing else records that relationship.

`build.js` otherwise passes waypoints/segments/alerts through close to
as-is (spread, not a hand-picked field whitelist), so most new fields on a
waypoint or segment just work without a `build.js` change.

Waypoints and segments each carry a stable `id` (documented in route.json's
own `schema_note_ids`):

- waypoint `id`: a slug of `name` (lowercase, underscores), generated once
  and never changed even if `name` is edited later.
- segment `id`: `<from_id>__<to_id>`.
- segments also carry `from_id`/`to_id`, which is the actual link to
  waypoints used for lookups (km, badges, waypoint cards).

`crossings.json` and `alerts.json` reference route.json entries by `id`
(`segment_ids`, `affects_waypoint_id`, `affects_segment_id`) -- never by
name or by numeric order/position.

**Why this split exists:** segments also keep plain `from`/`to` text fields,
and those are what the app actually displays for a leg's title -- edit them
freely, same as `name`. Earlier, segments matched waypoints by comparing
`from`/`to` text directly against `waypoint.name`, so renaming a waypoint in
one place silently broke the link everywhere else it was referenced. The
`id`/`from_id`/`to_id` fields exist purely to make that link immune to
renames; `from`/`to`/`name` stay freely editable without any risk of
breaking a lookup.

When adding a new waypoint, segment, or crossing: give it an `id` following
the same convention (slug for a waypoint, `from_id__to_id` for a segment),
and wire `crossings.json`/`alerts.json` to it by `id`. Then run
`node build.js`.

## UI structure

- Bottom tab bar, 3 tabs: itinerary (default), alerts (red badge = count of
  unresolved alerts), crossings.
- Itinerary: the full route in order, alternating waypoint cards and leg
  cards -- no hand-picked subset, no gap notes (that was pilot-only scope).
- Compact badge pills on every card (hut/crossing type, shop size, power,
  distance, ascent/descent, lake, shelter presence, alert flag) -- shown
  unconditionally, no filtering.
- Detail overlay: drilling into any card opens a full detail page. Nested
  JSON objects/arrays render generically as tables via `renderTable()`
  rather than being hand-formatted per field, so new JSON fields show up
  automatically without touching the render code.
- Alert banner at the top of the itinerary tab surfaces the first alert
  prominently, separate from the ordinary reference content.
- Print/PDF: a `@media print` stylesheet (browser-native print, no JS
  library) hides the tab bar and buttons and prints the itinerary/alerts/
  crossings content -- already implemented, not a roadmap item.

## Workflow

1. Edit `data/route.json` / `data/alerts.json` / `data/crossings.json` / any
   `data/boat_*.json` / `data/bus_*.json`.
2. Run `node build.js` (requires Node.js).
3. Open the regenerated `index.html` directly in a browser to check.
